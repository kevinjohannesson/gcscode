# Roadmap and project-status doc

**Status:** Approved (2026-04-27)

## Context

After A1, A2, A3, and B1 shipped, the phase plan exists in three places: this agent's session-memory file, ADR-0003's `## Follow-ups` section, and the `docs/specs/` + `docs/plans/` filename pattern. None is a single discoverable surface for "where are we now / what's next." Newcomers — human or agent landing on the repo — have to assemble the picture from several files. ADR-0003's follow-ups is the closest, but it lives inside an ADR (mixes retrospective with that ADR's specific decisions) and isn't where a fresh reader would naturally look.

This iteration creates `docs/roadmap.md` as the canonical start-here status surface, plus a small `CLAUDE.md` pointer.

## Goals

- A single-page `docs/roadmap.md` covering:
  - **Phase plan.** Phase A, Phase B, Phase C with sub-iteration status using GitHub-style checklists (`- [x]` shipped, `- [ ]` planned/deferred), with the deferral trigger noted inline.
  - **Feature plugins.** First-party plugins planned for the app, split into **Coming** (committed) and **Considering** (speculative).
  - **Maintenance protocol.** Explicit rules for when and how the doc is kept current — including the rule that user-mentioned ideas during chat should be propagated here.
- A new bullet in `CLAUDE.md` `## Further reading` pointing at the roadmap as the start-here doc.

## Non-goals

- **No duplication of `docs/out-of-scope.md`.** Roadmap and out-of-scope are complementary views: roadmap is by phase + iteration status, out-of-scope is by topic + deferral trigger. Both are kept; neither replaces the other.
- **No duplication of ADR rationale.** Roadmap links to ADR-0003 (or future ADRs) for "why".
- **No duplication of spec/plan content.** Roadmap links to specs; specs stay the source of truth for iteration detail.
- **No status line at the top of `CLAUDE.md`** duplicating the roadmap. Single source of truth = roadmap.
- **Not a full project status dashboard** (tooling, CI, dependencies, infrastructure). Out of scope for this iteration; those concerns live in `docs/out-of-scope.md` or their own files if/when they merit one.
- **No automated generation.** Maintenance is manual, mirroring how `docs/out-of-scope.md` and ADR-0003 retrospectives are kept current today.
- **No status badges, shields, Mermaid/graphviz diagrams.** Lean text only; the phase plan is a list, not a graph.

```md
# Roadmap

Single-page status: where the gcscode plugin architecture is and what's planned. Iteration detail lives in `docs/specs/`, plans in `docs/plans/`, rationale in `docs/decisions/`, deferral detail (organized by topic) in `docs/out-of-scope.md`. This file is the start-here "where are we now / what's next" view.

## Phase plan

The plugin architecture grows in phases: **A** (contribution kinds), **B** (lifecycle and cleanup), **C** (cross-cutting capabilities). Each phase contains numbered iterations.

### Phase A — Contribution kinds

- [x] **A1: Status bar item** — `host.registerStatusBarItem` + footer rendering. Spec: [`specs/2026-04-26-phase-a1-status-bar.md`](specs/2026-04-26-phase-a1-status-bar.md)
- [x] **A2: Commands** — `host.registerCommand` + `executeCommand` (host- and registry-side). Spec: [`specs/2026-04-26-phase-a2-commands.md`](specs/2026-04-26-phase-a2-commands.md)
- [x] **A3: Keybindings** — `host.registerKeybinding` + a shell-level keyboard dispatcher. Spec: [`specs/2026-04-26-phase-a3-keybindings.md`](specs/2026-04-26-phase-a3-keybindings.md)
- [ ] **A4+: more contribution kinds** — menus, palette entries, sidebar tree views all candidates. Trigger: a real consumer needs the surface. See [`out-of-scope.md`](out-of-scope.md).

### Phase B — Lifecycle and cleanup

- [x] **B1: Deactivate orchestration** — `registry.deactivate(pluginId)` iterates subscriptions LIFO with error resilience. Spec: [`specs/2026-04-26-phase-b1-deactivate-orchestration.md`](specs/2026-04-26-phase-b1-deactivate-orchestration.md)
- [ ] **B2: Plugin enable/disable + reactive plumbing** — adds runtime `enabled` state per plugin and the reactive plumbing so mounted UI reflects state changes. Trigger: a "disable plugin" UI or any visible state-change need.
- [ ] **B3: Dev-time hot module reload** — Vite HMR boundary that re-imports a plugin module on edit and replays activate. Trigger: plugin-author iteration friction.
- [ ] **`Plugin.deactivate?()` hook** — optional plugin-side hook for non-disposable / async cleanup. Split off from B1 by design. Trigger: first plugin needing it (named on-deck consumer: SITL listener — see Feature plugins below).

### Phase C — Cross-cutting capabilities

- [ ] **Phase C scope** — TBD. ADR-0003 sketches host namespacing (`host.commands.register(...)`) once the flat surface exceeds 5–7 methods, plus events, settings, themes, and i18n as real consumers pull on them. Re-scope when a feature plugin pulls on it.

## Feature plugins

The first-party plugins planned for the app. Each is a future consumer of the plugin architecture; some are named triggers for deferred phase work.

### Coming (committed — will ship)

- [ ] **SITL listener** — software-in-the-loop data-feed listener. First consumer of `Plugin.deactivate?()` hook (will hold a connection that needs explicit close); likely first to want a Phase C streaming/connection service.
- [ ] **Map** — geographical view + selection state. Likely fits the existing view contribution kind; may surface a need for shared map state.
- [ ] **Video feed** — live video stream display. Likely a Phase C streaming-source consumer alongside SITL.

### Considering (not yet committed)

- [ ] **Road scanning** — _description TBD_

## Maintenance

This doc is updated alongside other per-iteration docs propagation (`out-of-scope.md`, ADR-0003 retrospective):

- **Iteration ships** → flip its checkbox, link the spec, and (optionally) reference the merge SHA.
- **Iteration deferred or rescoped** → update its line + trigger, leave checkbox unchecked.
- **New deferral surfaces during a spec** → add an unchecked item with its trigger.
- **User mentions a feature idea offhand during planning chat** → propose adding it here. Default to **Considering**; escalate to **Coming** when the user signals commitment ("we'll need", "100% in", "definitely going to be").

The doc is a manually-maintained list, not a generated artifact. When in doubt about whether something belongs here, prefer the lighter touch — a one-line entry that links somewhere richer is better than re-stating that richer source.
```
